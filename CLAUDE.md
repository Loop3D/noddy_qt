# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

This is the C source code for **Noddy**, a geological forward-modelling application originally
developed at Monash University (first released 1994, this snapshot dated 2009). Noddy builds a 3D
geological model from a user-specified sequence of tectonic/depositional **events** (stratigraphy,
folds, faults, unconformities, shear zones, dykes, plugs, strain, tilt, foliation, lineation,
imported blocks, etc.), voxelises the result into a block model, and forward-calculates gravity and
magnetic anomalies from it.

The GUI is built on **XVT** (XVT Software's cross-platform C toolkit, XVT-Design/XVT-Architect),
which supported Windows, Mac, Motif/X11 (XOL/MTF) and SunOS from a single codebase via the `XVTWS`
window-system macro. XVT is a commercial/legacy SDK and is **not included in this repository** —
none of the `xvt*.h` headers or XVT libraries are present, so this source tree cannot be compiled
as-is. Treat this repo as a reference/archival source tree for reading, porting, and analysis rather
than something you build and run directly, unless the XVT SDK has been supplied separately.

## Build

- `MAKEFILE` is a relic from **1994** ("Generated by XVT-Design 2.01.03 ... Configured for
  XVT/XOL (SunOS)"). Its object-file list (`fold.o`, `fault.o`, `layers.o`, `strat.o`, `dyke.o`,
  `plug.o`, `tilt.o`, `reporter.o`, etc.) references source files that **no longer exist** in this
  tree — the code has since been reorganised (e.g. per-event logic now lives in `eventlib.c`,
  `events.c`, `ellFault.c`, `block.c` instead). Do not trust the makefile's file list; if you need
  to reconstruct a build, derive the object list from the `.c` files actually present and their
  `#include` dependencies instead.
- There is no modern build system (no CMake/Visual Studio project/etc.) in this tree.
- Later headers (`nodInc.h`) were regenerated by **XVT-Design 4.5** (1998), confirming the app was
  actively maintained well past the original 1994 makefile.

## Running (once built)

`main()` lives in `noddy.c`. It has two modes:

- **Interactive/GUI**: no recognised batch flags on the command line → calls
  `xvt_app_create(..., task_eh, &xvt_config)` and runs the normal windowed application (event
  handler `task_eh` in `noddy.c`, main history-builder window driven by `windowEventHandler`).
- **Batch/CLI**: if invoked with a `.his` (history) file or a `-h`/`-?` flag and no `-fn`/`-font`
  option, `main()` calls `batchNoddy()` in `batchNod.c`, which loads a history file via
  `Load_status()` and calls `performBatchOperations()`.

Batch usage (see `displayUsage()` in `nodLib3.c`):
```
noddy <history_file> [options]

  -h history_file
  -o output_file
  -b block_file            (see -anomAndBlock, -anomFromBlock)
  -block                   produce a block model of the geophysical area
  -dicer                   same as -block but in Dicer format
  -linemap                 produce a geological linemap diagram
  -section                 produce a geological section diagram
  -geoBlock                produce a geological block diagram
  -anom                    produce gravity and magnetic anomaly files
  -anomAndBlock            -anom plus a block file
  -anomFromBlock           -anom computed from a specified block file
  -profile                 force an anomalies profile to be produced
  -image                   force an anomalies image to be produced
  -parallel                run on parallel machine, if available (non-_MPL builds only)
  -help / -?               show usage
```
Any argument containing `.his` is treated as the history file regardless of position.

## Architecture

### Event-history model (the core domain concept)
A model is defined as an ordered list of **events** (`OBJECTS` enum in `nodStruc.h`):
`STRATIGRAPHY, FOLD, FAULT, UNCONFORMITY, SHEAR_ZONE, DYKE, PLUG, STRAIN, TILT, FOLIATION,
LINEATION, IMPORT, STOP, GENERIC`. Each event type has its own parameter struct in `nodStruc.h`
(`FOLD_OPTIONS`, `FAULT_OPTIONS`, `UNCONFORMITY_OPTIONS`, `DYKE_OPTIONS`, `PLUG_OPTIONS`,
`STRAIN_OPTIONS`, `TILT_OPTIONS`, `FOLIATION_OPTIONS`, `LINEATION_OPTIONS`, `IMPORT_OPTIONS`,
`GENERIC_OPTIONS`) hung off a generic `OBJECT` node (`next`, `shape`, `options`, ...) that forms a
linked list representing the history. Events are applied in sequence, each perturbing the geometry
established by the events before it (e.g. a fault event displaces/offsets everything deposited
earlier).

- `events.c` / `eventlib.c` / `ellFault.c` / `unEvents.c` — the kinematic math and UI logic for each
  event type (e.g. `fold()` in `events.c` computes the fold displacement field; `eventlib.c` holds
  the corresponding options-dialog logic such as `createEventOptions`, `updateFaultOptions`,
  `updateDykeOptions`, etc., one set per event type).
- `getEvent.c` / `putEvent.c` / `getHist.c` / `getStat.c` / `getput.c` / `readLine.c` — read/write the
  `.his` history file format (plain text; see `HISTORY_FILE_EXT` in `noddy.h`), including per-event
  `load*`/versioned parsing (`loadFold`, `loadFault`, `loadDyke`, ... in `getEvent.c`) driven from
  `readInHist()` in `getHist.c`.
- `builder.c` / `rmbuild.c` / `nodGraph.c` / `lineEvnt.c` / `mainMenu.c` / `lineMenu.c` — the
  drag-and-drop "history builder" window where users assemble the event sequence visually
  (icon/tool IDs defined in `builder.h`).

### Voxelisation / block model
- `block.c` — computes the 3D voxel block model from the event history (`calcBlockLayer`,
  `calcBlockDiagram`, `calcBlockSection`, `calcBlockImageValueData`, `calcDeformRemanence`,
  `calcDeformAnisotropy`, `calcAlterationZone`) and reads/writes Noddy block files
  (`writeBlockFile`, `write3DBlockToFile`, `read3DBlockFromFile`, headers via
  `writeBlockHeaderToFile`/`readBlockHeaderFromFile`).
- `DoBlock.c` — 3D rendering/preview of the block diagram (`renderBlockDiagram`,
  `drawBlockDiagramFrame`, `drawBlockBoreHoles`).
- `block3.c` / `lblock3.c` / `DoSect.c` / `DoSectLn.c` / `DoLMap.c` / `DoLMapLn.c` — section, line-map
  and borehole views/diagrams derived from the block model.
- `dotmap3.c` / `ldotmap.c` / `ldotmap2.c` / `smothmap.c` / `mcontr.c` — map/contour generation and
  smoothing over computed grids.

### Geophysics forward modelling
- `Macmag3.c` / `geophy.c` / `magCalc.c` / `calcanom.c` / `fft.c` — gravity/magnetic anomaly forward
  calculation from the voxel block model, both in the spatial and spectral (FFT) domains (see
  `wrapping.h` for the padding/wrapping schemes — RAMP/FENCE/SET — used to extend the model edges
  before spectral calculations). `GEOPHYSICS_OPTIONS` in `nodStruc.h` controls susceptibility units,
  field parameters (inclination/declination/intensity, fixed vs. variable field), calculation
  method (spatial/spectral/spatial-full) and padding.
- `geosoft.c` / `ERMapper.c` / `vulcan.c` / `xyzgen.c` / `xyzConvt.c` — import/export to third-party
  geoscience formats (Geosoft grids, ER Mapper, Vulcan, XYZ point data).
- `dxf*.c` / `dxf*.h` — DXF import/export for surfaces and volumes (used for e.g. digitised
  unconformity/fault surfaces as an alternative to the analytic `FLAT_SURFACE` type).

### 3D rendering
- `3d*.c` / `3d*.h` (`3dRender.c`, `3dSolids.c`, `3dSurf.c`, `3dClip.c`, `3dCamera.c`, `3dHermit.c`,
  `3dTextur.c`, ...) — a self-contained software 3D renderer (camera, clipping, lighting, texture
  mapping, solids/surfaces) used for the interactive 3D view and chair diagrams, independent of the
  XVT GUI toolkit itself.

### GUI layer (XVT)
- `nodInc.h` — machine-generated by XVT-Design; purely resource IDs/constants (window IDs, control
  IDs, menu tags) for every dialog. Not meant to be hand-edited — regenerate via XVT-Design if the
  UI layout changes.
- The ~70 `*win.c` files (`foldtwin.c`, `falttwin.c`, `dyke...`, `stratwin.c`, `sectwin.c`,
  `volwin.c`, etc.) are one dialog/window each, matching the options-window resources in
  `nodInc.h`; they read/write the corresponding `*_OPTIONS` struct from `nodStruc.h`.
- `noddy.c` — XVT application entry point and task-window event handler (`task_eh`); also owns
  licensing gate (`checkLicence()`) shown before the main window is created.
- `batchNod.c` — non-interactive entry point (see Running, above).
- `nodLib1.c` / `nodLib2.c` / `nodLib3.c` — large grab-bag libraries of shared helpers (memory
  manager `memManagerAdd`/`memManagerFree`*, licensing, window-position persistence,
  `processCommandLine`/`displayUsage`, profile/stratigraphy helpers, topography overlay).

### Parallel/remote variant
Several core files (`noddy.h`, `nodStruc.h`, `events.c`, `getHist.c`, etc.) carry a header warning
that they are shared between the standard sequential build and a **MasPar parallel version**
(`_MPL` macro) and a **remote** execution mode (`remProc.c`, `LOCAL_REMOTE_WINDOW`,
`REMOTE_CMD_STATUS` enum in `nodStruc.h`, `-parallel` batch flag). When editing these shared files,
keep changes compatible with the `_MPL`-guarded code paths even though the MasPar-specific
implementation is not part of this tree.

## Working in this codebase

- This is K&R/ANSI-transitional C: many functions are declared both with prototypes
  (`#if XVT_CC_PROTO`) and old-style K&R parameter lists (`#else`) for compatibility with older
  compilers — follow the existing dual-form pattern when changing a function signature (both
  branches must stay in sync in `fcnProto.h`/`nodLib3.c` and at each definition site).
- Struct/enum layouts in `nodStruc.h` are serialized directly to/from the `.his` text format by
  `getEvent.c`/`putEvent.c` with explicit per-field version checks (`VERSION_TOLERANCE`,
  `VERSION_NUMBER` in `noddy.h`) — adding/reordering fields requires corresponding load/save and
  version-gating changes, not just the struct edit.
- Window-system-specific behaviour is selected at compile time via `XVTWS == MACWS / XOLWS / MTFWS /
  WINWS / WIN16WS / WIN32WS`, not runtime checks — grep for `XVTWS` when touching anything
  window-system-sensitive (pen sizes, font sizes, resize handling, memory allocation on Mac).
